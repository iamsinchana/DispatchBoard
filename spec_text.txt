It consolidates orders from multiple sales channels (WhatsApp, Email/PO, e-commerce marketplaces, and direct dealer orders) into a single Kanban-style board visible on a touchscreen tablet on the packing floor.The system eliminates packaging and dispatch errors by making every pending order, its line items, fulfillment status, and payment state visible to the entire floor team in real time.Problem StatementThe current order process is fragmented across WhatsApp messages, email POs, and marketplace dashboards. Packaging errors occur because there is no single source of truth for what needs to ship, how much of each item has been shipped, and whether payment has been collected for cash-and-carry orders.Core Design PrinciplesCustomer name is the primary identifier, not order numbers. The floor team thinks in customer names.One board, one screen. No switching between dashboards or channels.Touch-first. The board runs on a tablet operated by packing staff with swipe and tap gestures.Partial fulfillment as a first-class concept. Orders can be shipped in multiple consignments.Payment gating is configurable per order. Regular distributors bypass payment; retail orders are blocked until paid.Workflow SpecificationBoard ColumnsThe board has five columns. Every order card moves left to right through these stages:ColumnPurposeEntry ConditionExit ConditionNewOrder just arrived from any channelAuto-created by intake parser or manual entryStaff reviews and advancesPaymentPayment gate or auto-passMoved from NewRegular: auto-pass. Internal UUIDs handle database relations, but the display ID shown on the board and used by floor staff is always the customer name. When a WhatsApp order comes in, the system fuzzy-matches the sender name against the customers table.Payment Gate LogicPayment behaviour is configurable per order based on customer type:Regular Distributor: Card badge shows "Credit — N/A". Card auto-passes through the Payment column. No interaction required from floor staff. Accounts team tracks payment separately outside this system.Non-Regular / Retail / New Dealer: Card badge shows "Payment Pending" in red. Card is hard-blocked at the Payment column. Cannot be moved to Fulfillment until someone taps "Mark as Paid". This prevents packing of unpaid cash-and-carry orders.New customers default to non-regular (payment required). Admin can promote them to regular via a settings panel.Design DecisionThe "Mark as Paid" button access level is configurable: initially open to all floor staff, but can be restricted to admin/sales roles in a later release via Supabase Row-Level Security policies.Partial Fulfillment and Sub-Order SpawningThis is the most important workflow in the system. Orders can be shipped in multiple consignments, and each partial shipment spawns a sub-order for the remaining items.The Sub-Order ModelWhen a packer opens the fulfillment modal and ships less than the full order, two things happen simultaneously:The original order card (e.g., "Ravi Electronics") moves to Shipped with only the quantities that were actually packed in this consignment.A sub-order card spawns automatically in the Fulfillment column carrying only the remaining line items. The sub-order is visually tagged with the parent order reference.Naming ConventionOrderDisplay IDContextOriginalRavi ElectronicsFirst order from this customerFirst sub-orderRavi Electronics — ARemainder after first partial shipmentSecond sub-orderRavi Electronics — BRemainder after second partial shipmentConcrete ExampleOriginal Order: Ravi Electronics, MumbaiItems: Product A × 50, Product B × 30, Product C × 50Fulfillment #1: Packer ships Product A × 25, Product B × 5, Product C × 25"Ravi Electronics" card moves to Shipped → Done with AWB BD-44821"Ravi Electronics — A" card spawns in Fulfillment with Product A × 25, Product B × 25, Product C × 25Fulfillment #2: Packer ships all remaining items from sub-order"Ravi Electronics — A" card moves to Shipped → Done with AWB BD-44835No further sub-order spawned (all items fulfilled)Sub-Order RulesSub-orders inherit: customer name, channel, priority from the parent order.Sub-orders carry only items with remaining quantity > 0. Fully shipped items are dropped.Sub-orders skip the Payment column (already cleared by the parent).Sub-orders enter the board at the Fulfillment column, not New.Sub-orders have priority auto-escalated (normal → high) to signal urgency.There is no limit to sub-order depth. Ravi Electronics — A can spawn Ravi Electronics — B if partially fulfilled again.Key InvariantAt any point in time, the sum of all shipped quantities across the original order and all its sub-orders must equal or be less than the original ordered quantity for each line item. The system must enforce this constraint at the database level.Technology StackArchitecture OverviewThe system uses a three-layer architecture with zero server management:LayerTechnologyPurposeFrontendNext.js 14 + React (PWA)Touch-optimized board UI, installable on tabletsBackendSupabase (Postgres + Realtime)Database, auth, realtime subscriptions, edge functionsHostingVercel (Mumbai edge)Frontend hosting, sub-50ms latency from BengaluruWhatsAppWati (WhatsApp BSP)India-focused WhatsApp Business API providerAI ParsingClaude API (Anthropic)Parse unstructured WhatsApp messages and PO emailsEmailIMAP Listener (cron)Poll orders inbox, extract PO dataWhy These Specific TechnologiesSupabase over Custom BackendThe engineering team is small. Supabase provides Postgres (for the relational order hierarchy), built-in realtime WebSocket subscriptions (so the tablet updates instantly when a WhatsApp order arrives), Row-Level Security (for future role-based access), and Edge Functions (for webhook handlers) — all without deploying or maintaining a server. The Supabase dashboard gives non-engineers visibility into the data.Wati over Twilio for WhatsAppWati is an India-focused WhatsApp Business Solution Provider. It is significantly cheaper than Twilio for Indian phone numbers, offers native template message support for order confirmations, and has a simpler Green Tick verification process. Wati fires webhooks on incoming messages, which hit a Supabase Edge Function.Claude API for Message ParsingWhatsApp messages and PO emails arrive in inconsistent formats — abbreviations, photos of handwritten lists, different conventions for quantities. A rules-based parser breaks the moment format varies. The Claude API handles ambiguity, extracts structured data (customer name, items, quantities, priority), and flags unparseable messages for manual review.Next.js PWA over Native AppA Progressive Web App installs on any Android tablet with a homescreen icon, full-screen mode, and no browser chrome — identical to a native app. It updates automatically through the web (no Play Store submissions), and the same URL works on the packing tablet, the floor TV (kiosk mode), and the admin phone. The frontend subscribes to INSERT, UPDATE, and DELETE events on these tables. When any intake channel creates an order or a packer changes a stage, every connected tablet receives the update within ~200ms via WebSocket.Intake ChannelsPriority Channel: WhatsApp via WatiWhatsApp is the highest-volume intake channel and should be implemented first.FlowSales person sends a structured message to the company WhatsApp Business number.Wati receives the message and fires a webhook to a Supabase Edge Function.The Edge Function sends the message text to the Claude API with a structured extraction prompt.Claude returns JSON: { customer_name, items: [{ product, qty }], priority, notes }The Edge Function fuzzy-matches customer_name against the customers table.If matched: creates order with the existing customer_id and their payment settings.If not matched: creates a new customer record (type: non_regular, payment_required: true) and the order.Order appears in the "New" column on all connected tablets instantly via Realtime.Claude API Prompt for ParsingThe Edge Function should use a system prompt that instructs Claude to extract structured data from informal WhatsApp messages. The prompt should handle variations like:"Ravi Electronics / A-50, B-30, C-50 / urgent" (structured format)"Send 50 nos of Product A and 30 nos of Product B to Ravi Electronics" (natural language)"Same as last order for Suresh Traders but double the PCBs" (requires context lookup — flag for manual review)Photo of handwritten PO (extract text via Claude vision, then parse)Wati ConfigurationCreate a Wati account with Green Tick verification for the business number.Set up a webhook URL pointing to the Supabase Edge Function endpoint.Configure template messages for order confirmation replies (sent back to the sender).Template example: "Order received for {{customer_name}}: {{item_summary}}. Tracking will be shared when shipped."Channel 2: Email / PO (Phase 2)A scheduled Supabase Edge Function (or Vercel cron) polls the orders inbox via IMAP every 2 minutes. New emails are sent to Claude API for extraction. The prompt handles both structured POs (PDF attachments with tables) and informal email orders. Unrecognized formats are flagged in the "New" column with a "Needs Review" badge.Channel 3: E-Commerce Webhooks (Phase 3)Amazon SP-API and Flipkart Seller API both support order webhooks. Each marketplace gets its own Supabase Edge Function endpoint. The customer name for marketplace orders is the marketplace name (e.g., "Amazon IN — FBA" or "Flipkart Marketplace"). Items are mapped from the marketplace SKU to the internal product name via a product mapping table (to be added in Phase 3).Channel 4: Manual EntryThe tablet board includes a "+ New Order" button that opens a full-screen form. Fields: customer name (autocomplete from customers table), channel (defaults to "direct"), priority, line items (product name + quantity), and notes. This is always available as a fallback for walk-in orders or phone calls.Frontend SpecificationTechnologyFramework: Next.js 14 (App Router) with React 18Styling: Tailwind CSS for utility-first responsive designState: Supabase JS client with realtime subscriptions (no additional state library needed)Gestures: Use a touch gesture library (e.g., @use-gesture/react) for swipe interactionsPWA: next-pwa plugin for service worker, manifest, and offline supportHosting: Vercel with Mumbai (ap-south-1) edge regionTablet Touch UX RequirementsThe board runs on a 10-inch Android tablet operated by packing staff. When a packer long-presses an order card in the Fulfillment column:Modal opens full-screen on tablet (not a small popover).Shows customer name, all line items with ordered quantities.Each item has large +/− buttons (56px) and an editable quantity field pre-filled with the max available."Ship All" button at top for one-tap full shipment."Clear All" button resets all quantities to 0.AWB / tracking number field with large on-screen keyboard.Summary bar at bottom: shows exactly what will ship and what will become a sub-order.Two-step confirmation: "Ship Partial" button, then "Yes, create sub-order and ship" confirmation.On confirmation: creates shipment record, creates sub-order (if partial), moves original card to Shipped.TV / View-Only ModeThe same URL with a "?mode=tv" query parameter enables view-only mode for the floor TV. In this mode: all cards are visible across columns, the board auto-scrolls through columns if content overflows, touch interactions are disabled, and the display auto-refreshes via Supabase Realtime. An audio ping plays when an urgent order arrives.Key API ContractsWati Webhook Payload (Incoming)POST /functions/v1/wati-webhookWati sends a JSON payload containing the sender phone number, message text, and optional media URLs. The Edge Function should extract the text field and pass it to Claude API for parsing.Claude API Parsing RequestThe Edge Function sends the WhatsApp message text to Claude with a system prompt instructing structured extraction. Expected response format:{ customer_name: string, items: [{ product: string, qty: number }], priority: "urgent" | "high" | "normal", notes: string, confidence: number }If confidence < 0.7, the order is created with a "needs_review" flag and appears in the New column with a yellow badge.Supabase Realtime SubscriptionThe frontend subscribes to changes on the orders table:supabase.channel('orders').on('postgres_changes', { event: '*', schema: 'public', table: 'orders' }, handleChange).subscribe()This covers INSERT (new orders from any channel), UPDATE (stage changes from packer interactions), and DELETE (archival). The same subscription pattern applies to order_items for quantity updates.Sub-Order Creation (Database Function)Implement the sub-order spawning logic as a Postgres function called from the frontend. This function should be atomic (wrapped in a transaction) to prevent race conditions:Accept: order_id, array of { product_name, qty_to_ship }, awb, courierValidate: for each item, qty_to_ship <= qty_orderedCreate shipment record and shipment_itemsCalculate remaining quantities per itemIf any item has remaining > 0: create sub-order with suffix, create order_items for remaining quantities onlyMove original order stage to "shipped"Return: { shipment_id, sub_order_id (nullable) }Implementation PhasesPhase 1: Core Board + Manual Entry (Week 1–2)Set up Supabase project (Mumbai
